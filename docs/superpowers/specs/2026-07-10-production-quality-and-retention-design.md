# 健身饮食小程序生产质量与留存改进设计

**日期：** 2026-07-10  
**范围：** 除微信授权登录、openid 获取和微信订阅消息发送外，修复首轮审计发现的问题，并实现已确认的核心产品改进。

## 1. 目标与边界

本轮把当前内部测试版提升到可进行正式小规模验证的状态。工作分为四条主线：消除生产安全和隐私风险；修正产品承诺与实际能力不一致；降低饮食与训练记录成本；提供简单、可解释的行为反馈。

本轮不实现真实微信授权、微信订阅消息、真实 AI 识别、会员付费、社区、排行榜或管理后台。现有手机号注册登录继续作为可测试入口；微信 mock 接口保持现状，不纳入本轮验收，也不新增相关入口。

## 2. 配置与生产安全

- `db_url` 和 `jwt_secret` 不再包含可用默认凭据。开发环境通过 `.env` 注入，并提供不含秘密的 `.env.example`。
- 非调试环境缺少数据库地址或使用弱 JWT 密钥时，应用启动失败并给出明确错误。
- 仓库中已经出现的数据库密码视为泄露凭据，必须由项目所有者在数据库服务商处轮换；代码只能阻止继续暴露，不能代替外部轮换。
- 密码校验改用恒定时间比较；保留现有密码数据兼容，不在本轮引入新的密码依赖或强制迁移全部用户。
- 验证码增加单次使用、过期清理、错误次数上限和进程内容量上限。单实例 MVP 继续使用内存存储；多实例部署前再切换 Redis。

## 3. 个人数据与账号生命周期

产品明确区分两个操作：

1. **清空健身数据**：保留登录账号、昵称、头像和完成基础功能所需的账号状态；删除饮食、训练、体重、自定义食物、自定义动作、营养目标、提醒、识别记录、上传文件和身体资料。
2. **注销账号**：先执行与清空健身数据相同的清理，再清除手机号、openid、密码、昵称、头像等可识别字段，将账号状态改为 `cancelled`，使现有 token 立即失效。

操作日志只保留不含个人内容的最小审计事件。数据库删除先提交，文件随后删除；文件删除失败记录错误但不恢复已经完成的数据库删除。前端文案准确说明实际效果，不再承诺不存在的数据导出能力。

## 4. 图片隐私与生命周期

- 头像继续作为公开资源；饮食照片和识别临时图片改为鉴权访问。
- 上传成功后统一解码并重新编码为安全的 JPEG 或 PNG，修正方向并移除 EXIF；不保留用户提供的原始文件内容。
- 上传记录继续校验所有权、用途、真实图片格式和大小。
- 临时识别图片在用户确认记录时根据“保存图片”选项转为永久或删除；超时临时文件由现有清理流程删除。
- API 返回可供前端显示的受保护资源路径；前端通过带 token 的下载/预览流程读取私有图片。

## 5. 产品能力与文案一致性

- 数据统计周期统一为 `7/30/90` 天，前端类型、筛选器、后端校验、统计测试和 PRD 同步。
- 提醒页面和入口暂时隐藏。数据库/API 可保留，但在没有微信订阅消息能力前，不对用户声称会收到系统通知。
- 拍照识别页面明确标注“模拟候选推荐”，说明结果不来自真实图片分析；用户必须选择食物并填写重量后才能保存。
- 账号页移除空的“账号安全”卡片和不可用的导出占位。

## 6. CSV 数据导出

增加一个鉴权导出接口，按 UTF-8 BOM CSV 返回当前用户的数据。为避免引入压缩包或 Excel 依赖，第一版导出一份纵向 CSV，使用 `record_type` 区分：

- 用户目标与身体资料
- 饮食记录
- 训练场次、动作和组
- 体重记录

每行包含稳定的公共列和该记录类型的 JSON `details` 字段。前端账号页提供“导出个人数据”按钮，在支持文件保存的平台保存并提示路径；平台不支持时给出明确提示。注销弹窗只有在导出功能可用时才提示用户先导出。

## 7. 降低饮食记录成本

### 7.1 最近吃过

添加饮食页展示当前用户最近使用的去重食物，按最后记录时间倒序。选择后复用食物来源、营养快照和最近一次有效份量，用户可修改日期、餐次和数量后保存。

### 7.2 复制上一餐

饮食页允许把最近一顿完整复制到选定日期和餐次。后端使用单事务创建新记录，复制食物与营养快照，但使用目标日期、当前时间或用户选择的时间；图片默认不复制，避免隐私文件和生命周期耦合。重复点击由前端提交锁防止，不增加复杂幂等基础设施。

## 8. 降低训练记录成本

创建训练 session 时，后端按用户和动作快照名称查找最近一次已完成训练中的同动作数据。新 session 的各组目标重量和次数优先使用上次相同组序号的实际值；不存在历史值时使用计划目标。

训练执行页明确展示“已带入上次记录”，用户仍可编辑。现有训练进度自动保存和继续训练流程保持不变。

## 9. 周总结与可执行反馈

新增规则型周总结接口，不接大模型。统计最近 7 个自然日并与前 7 日比较，返回：

- 饮食记录天数、平均热量、蛋白质达标天数
- 训练次数和总容量变化
- 体重首末变化
- 最多三条按固定优先级生成的行动建议

建议只陈述数据支持的事实，例如“本周有 3 天蛋白质达到目标”或“连续记录不足 3 天，先从每天记录一餐开始”。数据不足时明确显示“记录更多数据后生成总结”，不提供医疗判断或自动修改目标。

首页只展示最重要的一条下一步行动；完整周总结放在数据页。连续记录天数只作为轻量反馈，不使用惩罚性断签文案。

## 10. 接口与错误处理

- 保持现有 `{code, message, data}` JSON 响应；CSV 下载是唯一的文件响应例外。
- 所有读取和写入继续以 `get_current_user` 和 `user_id` 过滤保证隔离。
- 批量复制、清空数据和注销在数据库事务内完成。
- 文件导出、上传和删除错误返回可执行提示；前端失败后恢复 loading 状态并保留用户输入。
- 不为单实例 MVP 新增队列、Redis、事件总线或通用仓储层。

## 11. 测试与验收

实施遵循测试先行，每个行为先看到失败测试，再写最小实现。自动检查至少覆盖：

- 生产配置拒绝源码默认秘密，开发 `.env` 可启动。
- 密码恒定时间校验和验证码错误次数/清理。
- 清空数据与注销的表级清理范围、文件删除及 token 失效。
- 私有图片所有权、公开头像、图片重编码和 EXIF 移除。
- `7/30/90` 统计范围。
- 最近吃过去重排序、复制上一餐的数据隔离和事务。
- 训练 session 带入上次实际重量和次数。
- 周总结在有数据、无数据和目标缺失时的稳定输出。
- CSV 只包含当前用户数据，并正确处理中文。
- 前端页面入口、状态和文案测试。

最终运行后端完整测试、前端完整测试、TypeScript 类型检查、H5 构建和微信小程序构建。微信授权登录、微信订阅消息、相机和真实设备文件保存列为人工或后续验收项，不阻塞本轮其余功能完成。

## 12. 分阶段交付顺序

1. 配置秘密、账号数据清理、图片隐私。
2. 统计周期和误导文案修正、CSV 导出。
3. 最近吃过、复制上一餐、训练历史值带入。
4. 周总结、首页行动提示和完整回归验证。

每一阶段必须能独立通过测试，不依赖后续阶段才能恢复现有功能。
