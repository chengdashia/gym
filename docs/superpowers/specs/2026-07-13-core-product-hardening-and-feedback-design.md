# 核心产品收口、稳定性与反馈闭环设计

## 目标

在不开发会员支付和真实 AI 识别的前提下，消除当前产品中的虚假能力承诺，修复关键数据一致性与安全问题，并将首页、训练结束和趋势统计改造成能够推动用户每日行动的完整反馈闭环。

## 范围与原则

- 本次隐藏拍照识别入口，但保留页面、API、后端 mock 和历史数据，后续再接入真实 AI。
- 本次完全隐藏前端会员身份、到期时间和相关入口，不开发支付与权益；后端字段保持兼容。
- 提醒设置继续保留，但仅保存计划，明确告知当前不会发送系统通知。
- 首次登录必须完成协议确认、头像、昵称和核心目标，其他身体资料采用渐进式补充。
- 优先复用现有接口、服务、组件与功能开关，不引入新依赖或为未来需求搭建抽象。
- 统计与趋势规则由后端统一计算，前端只展示结果，日期第一版统一按 `Asia/Shanghai` 处理。

## 一、功能收口与真实性

### 拍照识别

- 在饮食首页、添加饮食等普通导航路径隐藏拍照识别入口。
- 使用现有 `feature-gates` 集中控制入口，不在页面散落常量判断。
- 普通用户无法从产品导航进入识别页；保留识别页面、API、mock provider、数据表和历史记录。
- 为功能开关增加单元测试，防止未接入真实服务前入口被误开放。

### 会员

- 移除前端“会员/普通用户”、会员到期时间、购买入口和权益说明。
- 保留数据库字段和现有接口响应字段，不增加数据库迁移。
- 会员字段不能影响本次任何普通功能的可用性。

### 提醒

- 页面文案明确说明：“当前仅保存提醒计划，不会发送系统通知。”
- 保存成功提示改为“提醒计划已保存”。
- 允许取消全部星期；开启提醒但没有选择星期时禁止保存，并提示至少选择一天。
- 不申请微信订阅消息权限，不增加定时任务或消息发送服务。

## 二、首次登录与渐进式资料

### 首次登录步骤

1. 完成微信或手机号登录。
2. 阅读并确认用户协议与隐私政策。
3. 上传或选择头像并填写昵称，两项均为必填。
4. 选择一个核心目标：减脂、增肌或维持健康。
5. 进入首页。

如果用户中途退出，下次登录继续尚未完成的步骤，不重复创建资料。已有完成引导的用户不得被重新导向首次登录。

### 渐进式补充

- 年龄、性别、身高、当前体重、目标体重和训练频率不阻塞首次进入。
- 用户使用营养推荐、体重趋势或训练推荐时，只提示补充该功能真正需要的字段。
- 不完整资料不影响手动饮食记录、手动训练计划和基础浏览。

## 三、数据一致性与安全

### 统一资料校验

- 为年龄、身高、当前体重和目标体重建立一份前端共享规则，并让输入限制与提交校验使用同一规则。
- 后端 Schema 使用对应范围作为最终信任边界。
- 可选字段为空时保持为空，失焦不得自动填入最小值。
- 营养推荐在必要字段缺失或超出公式适用范围时返回明确的业务错误和所缺字段。

### 自定义食物事务

- 新增一个后端事务接口，在同一事务内创建自定义食物并创建饮食记录。
- 任一步失败时全部回滚，不留下孤立食物或半成品记录。
- 前端自定义食物页面改用该接口；原有独立创建接口继续服务“只创建食物”的场景。

### 状态与错误

- 搜索无结果、网络失败和服务端失败使用不同的用户状态与文案。
- 开始训练、保存训练组和结束训练保持按钮级重复提交保护；服务端对结束训练提供幂等行为。
- 重复请求不得产生重复训练结束、重复饮食记录或重复统计累计。

### 权限与上传

- 逐一验证饮食记录、自定义食物、训练计划、训练场次、体重记录和饮食方案的读取及修改都限定为当前用户资源。
- 针对跨用户读取、修改和删除增加自动化测试。
- 图片上传校验文件大小、声明 MIME、扩展名和实际可解码图片内容，服务端生成文件名。
- 对登录、上传和数据导出采用项目当前部署能力可支持的最小限流方案。
- 测试环境允许 mock 微信登录；生产配置启用 mock 或使用默认密钥时应用启动失败。

## 四、行动型首页

### 今日主要行动

首页根据状态只突出当前最重要的一项，优先级如下：

1. 补充当前功能所需资料。
2. 生成每日营养目标。
3. 记录当前时段尚未记录的餐次。
4. 继续未完成训练。
5. 开始今日计划训练。
6. 在适合的提示条件下记录体重。
7. 所有主要任务完成后显示今日完成反馈。

### 今日营养

- 展示已摄入、剩余或超出热量，以及碳水、蛋白质和脂肪完成情况。
- 剩余值不得显示为负数；超标时单独显示“已超过目标”。
- 优先给出可行动信息，例如剩余热量和蛋白质缺口。

### 今日训练

- 明确区分无计划、休息日、待开始、进行中、部分完成和已完成。
- 进行中直接进入继续训练；完成后进入训练总结。

### 体重与本周反馈

- 有足够数据时显示当前体重、7 日移动平均和周期变化。
- 数据不足时不生成趋势，显示仍需补充的记录提示。
- 本周反馈显示饮食记录天数、训练完成次数、体重记录次数、连续记录天数及与上周变化。
- “有效行为日”定义为当天至少记录一餐、完成一次训练或记录一次体重；同日多次行为只累计一天。

### 接口边界

- 扩展现有首页聚合接口，一次返回主要行动、营养、训练、体重与周反馈所需数据。
- 首页页面只负责展示和跳转，趋势、连续天数及周期比较由后端服务计算。
- 某个统计子模块失败时，其他首页模块仍可展示；响应为失败模块提供可识别的空状态。

## 五、训练总结

训练完成后展示：

- 训练名称、完成时间和总时长；
- 完成动作数、完成组数和计划组数；
- 本次总训练容量及与同类上次训练的变化；
- 每个动作的实际重量、次数和完成组数；
- 未完成内容的明确标记；
- 返回首页和查看训练历史两个后续动作。

当某动作全部计划组完成且实际次数达到目标时，可以提示下次尝试小幅增加重量，但不得自动修改训练计划。训练已经完成而总结加载失败时，重新加载总结，不允许再次提交结束训练。

## 六、体重趋势与周反馈规则

### 体重趋势

- 7 日均线使用最近 7 个自然日中的有效体重记录。
- 同一自然日多次记录时，趋势采用当天最后一次记录。
- 少于 3 个有效记录时不计算趋势。
- 趋势同时提供原始体重点和移动平均，前端不得只显示均线。
- 周期比较使用相邻两段的有效均值，不以单日涨跌下结论。
- 体重变化不自动修改营养目标。

### 周反馈

- 自然周内统计有饮食记录的天数、营养达标天数、完成训练次数、计划训练完成率、体重记录次数和连续记录天数。
- 营养达标定义为当天总热量位于目标值的 85%–115%。目标缺失或为零时不参与达标统计。
- 与上周比较使用相同口径；数据不足时明确返回数据不足，不生成推断性建议。

## 七、错误处理和测试

自动化测试至少覆盖：

- 功能开关隐藏拍照识别和会员展示；
- 提醒无星期校验及允许全部取消；
- 首次登录中断恢复、必填头像昵称、只要求核心目标；
- 可选资料空值、前后端边界一致；
- 自定义食物事务成功与回滚；
- 所有关键资源的跨用户访问被拒绝；
- 非图片、伪造 MIME、超限图片和无效图片上传失败；
- 重复结束训练的幂等行为；
- 同日多次体重、少于 3 次记录、跨周/月/年边界；
- 部分完成、取消及完整训练总结；
- 零营养目标、目标超标、85% 和 115% 边界；
- 连续记录中断与恢复；
- 首页单个统计模块失败时的降级展示。

验证必须包括前端单元测试、前端微信小程序构建、后端单元及集成测试。涉及小程序原生行为的登录、头像选择和页面跳转还需在微信开发者工具中手动验证。

## 八、交付顺序

1. 功能收口与真实性修复。
2. 首次登录、数据一致性、安全和事务修复。
3. 首页聚合接口与行动型首页。
4. 训练总结、体重趋势和周反馈。

每个阶段都必须保持可构建、可测试，且不得依赖后续阶段才能正常使用。

## 不在本次范围

- 会员套餐、支付、权益、续费和退款。
- 真实 AI 图片识别与识别计费。
- 微信订阅消息授权、定时任务和通知发送。
- 用户自定义时区。
- 自动调整营养目标或自动修改训练计划。
- 医疗、诊断或治疗建议。
