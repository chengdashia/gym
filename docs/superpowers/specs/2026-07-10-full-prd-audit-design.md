# 健身饮食记录小程序完整需求审计与补齐设计

**日期：** 2026-07-10  
**需求来源：** `docs/健身饮食记录小程序_完整汇总文档.md`

## 1. 目标与验收口径

本轮以汇总文档中的“第一阶段必须实现”“成功标准”“前后端验收标准”和 API 清单为验收依据，检查现有 uni-app + Vue 3 前端及 FastAPI + MySQL 后端。已经实现的功能需要核对真实流程和边界条件；缺失功能需要补齐；发现的缺陷需要在根因处修复并留下可重复验证。

登录主流程采用当前手机号登录/注册。微信授权登录保留已有接口和扩展能力，本轮不把它切换为主入口。未登录用户可以浏览无个人数据的首页，任何需要读取或写入个人数据的入口统一跳转登录页。成功登录后必须持久化 token，冷启动时先校验已有 token，不能主动清除有效登录态。

涉及数据库字段或表结构的修改通过 Alembic 迁移直接同步当前配置的数据库，不创建隔离测试数据库。端到端联调使用专用测试账号，完成后清理该账号产生的业务数据。纯计算和不依赖数据库的逻辑仍保留自动化单元测试。

## 2. 实现策略

保持现有前后端架构和页面视觉体系，只在当前模块内做必要修改。优先复用已有 API、Pinia store、公共组件和服务函数，不进行无关的目录重构，也不为单一实现增加抽象层。

后端负责身份校验、用户数据隔离、输入约束、营养快照、训练排期、session 状态、统计聚合和个人数据清理。前端负责表单交互、页面状态、登录拦截、倒计时体验和数据展示。所有关键规则以后端为最终裁决，前端同步做即时校验以改善用户反馈。

## 3. 功能设计

### 3.1 登录、首次引导与账户状态

- 保留手机号注册、手机号密码登录和图形验证码。
- 恢复 token 与用户摘要的持久化；启动时读取缓存并通过 `/users/me` 校验。
- 网络暂时失败不能被误判为 token 失效；只有明确的 401 才清除登录态。
- 新用户完成注册后进入协议确认、基础资料、营养目标推荐与确认流程。
- 注销账号成功后立即清除前端 token、用户、饮食、训练等本地状态。
- 微信登录接口继续保留，但不在主页面增加微信授权入口。

### 3.2 首页

- 饮食卡展示热量与三大营养素、餐次记录数量和“添加饮食”入口。
- 未设置营养目标时显示明确提示并可进入目标设置。
- 训练卡正确区分无计划、休息、未开始、进行中、暂停和已完成。
- 任意计划遗留的未完成 session 优先显示“继续训练”，不能因为切换当前计划而丢失入口。
- 体重卡展示当前值、目标值、差距、最近记录时间，并支持记录体重和进入趋势页。
- 首页重新显示时刷新聚合数据，确保饮食、训练、体重写入后即时反映。

### 3.3 饮食、食物与拍照识别

- 系统食物和当前用户自定义食物继续统一搜索，后端保持用户隔离。
- 创建自定义食物后继续完成数量、餐次、日期、时间填写，并创建饮食记录，而不是只保存食物后退出。
- 搜索添加、自定义添加和拍照识别都支持有效的克数或份数；数量必须大于零。
- 支持选择历史日期补录；记录编辑支持日期、时间、餐次、克数或份数及删除。
- 后端对单位和数量组合做一致性校验，避免缺少克数、缺少份数或零数量形成无效记录。
- 拍照识别失败时保留手动搜索入口；保存记录前校验候选食物和重量。
- 用户选择“不保存图片”时，饮食记录不绑定图片，临时上传文件在识别完成或定期清理路径中删除；选择保存时将文件标记为非临时。
- 上传接口校验真实图片内容、文件大小和允许的用途，不能只依赖扩展名。

### 3.4 饮食目标与体重联动

- 推荐目标、手动修改目标和首页目标进度保持现有接口。
- 当目标缺失时，store 使用明确的空目标状态，不把 `null` 当作完整目标对象。
- 根据最近体重变化提供“建议重新评估饮食目标”提示；提示只负责引导到目标页，不自动修改目标。

### 3.5 训练计划、动作与模板

- 模板创建和完全自定义创建继续使用同一计划编辑页。
- 增加自定义动作创建入口，创建后可立即加入当前训练日。
- 保存前验证计划至少包含一个训练日；非休息训练日至少包含一个动作；按周排期的星期不可重复；动作组数、次数、重量和休息时间必须有效。
- 后端重复执行同样的计划结构校验，防止绕过前端创建不可执行计划。
- 激活新计划不删除旧计划；历史 session 保留快照数据。

### 3.6 今日训练、训练执行与历史

- 修复顺序循环推进：完成当前训练后推进到下一可训练日，并正确跨日期工作；休息日不能让顺序计划永久停住。
- 按周排期继续按 ISO 星期一至星期日匹配。
- 同一用户只允许一个进行中或暂停中的 session；重复开始时返回已有 session 供继续。
- session 更新支持完成和取消完成某一组，完整重算动作完成组数和 session 容量，不能只统计本次请求片段。
- 完成一组后按设置自动启动倒计时；支持暂停、继续、跳过、临时增减时间、手动开始和关闭自动倒计时。
- 倒计时结束提供页面状态、震动和声音提示；不支持声音的平台安全降级为震动和页面提示。
- 退出训练提供继续训练、保存进度、放弃训练和标记完成。标记完成前保存当前组数据。
- 训练历史展示日期、名称、时长、容量、动作和每组实际数据，并支持删除记录。

### 3.7 体重记录

- 记录表单包含体重、记录日期、记录时间和备注。
- 展示历史记录并支持编辑、删除；所有操作继续使用现有 CRUD API。
- 新增、修改或删除后刷新首页、体重趋势及本页历史。
- 最新体重以时间排序后的有效体重记录为准；没有记录时才回退到资料中的当前体重。

### 3.8 数据统计

- 7、30、90 天切换继续返回连续日期，缺失饮食或训练日期填零。
- 饮食统计展示热量、三大营养素和热量目标完成率。
- 训练统计展示训练次数、时长、容量，并补充每个动作的实际重量趋势。
- 体重统计展示趋势、目标差距和所选周期变化。
- 统计与首页使用相同的日期边界和软删除过滤规则，避免同一天末尾数据遗漏。

### 3.9 提醒、合规与个人数据

- 饮食、训练、体重提醒支持启用、时间和星期设置，并校验时间与星期范围。
- 用户协议、隐私政策、图片识别用途、数据导出占位入口继续保留。
- “删除个人数据”清理饮食、体重、训练、计划、自定义食物、自定义动作、营养目标、提醒、资料、识别记录和上传文件，并删除本地上传实体文件；用户账号本身保留。
- “注销账号”执行个人数据清理并把账号标记为已注销，现有 token 随即失效；相同手机号不能静默恢复已注销账号。

## 4. 数据流与一致性

每次写操作采用“前端校验 → 后端鉴权和业务校验 → 单事务写入 → 返回最新资源 → store/页面刷新”的流程。首页、数据页和历史页不依赖前端自行累加，而是重新读取后端聚合结果。

所有按日查询统一使用半开区间 `[当天 00:00:00, 次日 00:00:00)`，避免 `23:59:59` 边界遗漏。训练容量只累计已完成组的 `实际重量 × 实际次数`。体重同一天存在多条记录时采用记录日期、记录时间和记录 ID 的确定性顺序选择最新值。

数据库迁移必须先检查当前 Alembic revision，再执行 `upgrade head`。如果模型与现有表结构不一致，新增迁移只包含本轮必需变化，不重建或清空业务表。

## 5. 错误处理

- 业务错误继续使用 `{code, message, data}` 结构。
- 参数格式和业务组合错误使用 `40001`；未登录或 token 失效使用 `40101`；越权不暴露资源是否存在；状态冲突使用 `40901`。
- 前端统一显示可执行的错误提示，并保证 loading、saving 和弹窗状态在失败后恢复。
- 只有明确认证失败才跳登录；普通网络错误保留当前页面和用户输入。
- 文件写入与数据库写入失败时清理已生成的孤立文件。

## 6. 测试与验收

### 6.1 自动检查

- 为营养计算、目标推荐、顺序/周排期、统计聚合辅助逻辑和训练容量重算增加小型自动测试。
- 后端接口联调直接连接当前配置数据库，使用唯一测试手机号和测试数据前缀，覆盖注册、资料、目标、食物、饮食、拍照识别、计划、session、体重、统计、删除数据和注销。
- 联调脚本无论成功或失败都执行测试数据清理，并输出每个业务闭环的结果。
- 前端执行 TypeScript 检查、H5 构建和微信小程序构建。

### 6.2 浏览器与小程序流程

- H5 浏览器验证登录/注册、首次引导、五个主页面、饮食三种添加方式、训练计划与执行、体重 CRUD、统计切换、提醒和合规入口。
- 浏览器验证页面非空、无框架错误覆盖层、无相关控制台错误，并检查移动视口下主要操作不被底部导航遮挡。
- 微信小程序生产构建完成后检查产物页面、tabBar 和平台条件编译；相机、震动和声音等平台能力在微信开发者工具中列为最终人工验收项。

## 7. 完成标准

完成时需要提供：需求覆盖清单、已修复缺陷及根因、数据库迁移结果、自动测试结果、H5 与微信小程序构建结果、关键浏览器流程结果，以及仍依赖真实微信环境的人工验收项。没有验证证据的功能不能标记为完成。
