# 健身饮食离线 Android APK 设计

## 1. 目标

将当前前后端分离的健身饮食项目改造成单机 Android APK。应用安装后不依赖服务器、MySQL、微信登录或互联网，用户可以在飞行模式下使用个人资料、营养目标、饮食记录、训练、体重、统计、照片留档和本地备份恢复功能。

第一阶段只面向 Android。复杂饮食方案、阶段调整和自动膳食计划放入后续阶段。

## 2. 已确认的产品决策

- 食物拍照不进行自动识别。照片只用于留档，食物由用户手动搜索和选择。
- 应用不需要登录。首次启动时填写个人资料，此后直接进入应用。
- 第一版提供完整的离线备份与恢复。
- 第一阶段迁移个人资料、营养目标、饮食、训练、体重、统计和备份；复杂饮食方案后续迁移。
- 应用不得依赖远程 API，并且不得申请联网权限或访问手机全部文件的权限。

## 3. 方案选择

采用“现有 uni-app 页面 + TypeScript 本地业务层 + Android SQLite”的架构。

不采用 APK 内嵌 Python/FastAPI。该方案会增加安装包体积、进程管理和 Android 兼容成本，也不能直接复用当前 MySQL 数据层。不采用 `uni.setStorageSync` 作为主数据存储，因为训练、饮食和统计数据需要关系查询、事务、索引及可靠迁移。

## 4. 总体架构

```text
uni-app Vue 页面
        ↓
本地 Service 接口
        ↓
TypeScript 业务规则
        ↓
Repository
        ↓
SQLite 数据库 + 应用私有图片目录
        ↓
ZIP 备份文件导出/恢复
```

页面尽量保留当前调用方式。现有 `dietApi`、`trainingApi`、`weightApi` 等模块不再通过 `uni.request` 请求 FastAPI，而是调用本地 Service。Service 处理业务规则，Repository 只处理数据库读写。页面不得直接拼接 SQL 或直接管理应用私有文件路径。

应用运行时不再需要 FastAPI、MySQL、JWT、微信登录或 `API_BASE`。原 Python 后端保留在仓库中作为既有规则和测试结果的参考，但不打包进 APK。

## 5. 本地数据设计

SQLite 数据库包含以下核心表：

- `local_profile`：固定单用户的个人资料和首次启动状态。
- `nutrition_goals`：热量、碳水、蛋白质和脂肪目标。
- `foods`、`custom_foods`：内置食物与用户自定义食物。
- `diet_records`：饮食记录及创建时的食物、份量和营养快照。
- `exercises`、`custom_exercises`：内置动作与用户自定义动作。
- `training_templates`、`training_plans`、`training_plan_days`、`training_plan_exercises`：训练模板和计划。
- `training_sessions`、`training_session_exercises`、`training_session_sets`：训练执行及历史记录。
- `weight_records`：体重历史。
- `uploaded_files`：应用私有目录中的照片相对路径、用途和状态。
- `app_meta`：数据库版本、种子数据版本和初始化状态。

数据库规则：

- SQLite 初始化时开启外键约束，涉及多表的保存和删除使用事务。
- 营养值和体重明确存储精度；业务计算采用统一的舍入函数，避免不同页面产生不同结果。
- 饮食记录保存营养快照。修改或升级内置食物库不得改变历史记录。
- 用户记录默认采用软删除；内部模板和不可恢复的临时文件可以物理删除。
- 数据库使用递增的 schema version 和顺序迁移。APK 升级只迁移数据，不清空数据库。
- 内置食物、动作和训练模板由版本化种子数据初始化。种子升级不得覆盖用户自定义内容。

## 6. 图片处理

拍照或选择图片后，应用将图片复制到应用私有目录，并在 `uploaded_files` 中保存相对路径。页面随后进入手动搜索食物流程，由用户选择食物、填写克数或份数，再保存饮食记录。

如果图片复制失败，用户可以不带图片继续保存记录。删除引用图片的记录时，只有确认该文件不再被其他记录引用后才删除物理文件。

应用移除识别中状态、置信度、模拟候选食物和识别日志。第一阶段不包含任何端侧 AI 模型。

## 7. 备份与恢复

导出文件格式：

```text
gym-backup-YYYY-MM-DD.zip
├── manifest.json
├── database.sqlite
└── images/
    ├── diet/
    └── avatar/
```

`manifest.json` 包含备份格式版本、应用版本、数据库版本、导出时间、图片清单和各文件校验值。

导出通过 Android 系统文件选择器让用户选择保存位置。恢复同样通过系统文件选择器选择备份文件，因此不申请访问手机全部文件的权限。

恢复流程必须：

1. 将备份解压至应用临时目录。
2. 验证 ZIP 路径安全、清单格式、版本兼容性、数据库完整性和文件校验值。
3. 自动保存当前数据的恢复点。
4. 关闭数据库连接并原子替换数据库与图片目录。
5. 重新打开数据库并执行必要的向前迁移。
6. 任一步骤失败时恢复原数据，且不得让应用在半恢复状态下继续运行。

备份只包含用户数据和用户保留的图片。内置种子数据仍保留在数据库备份中，以保证恢复结果与导出时一致。

## 8. 第一阶段功能范围

第一阶段包含：

- 首次启动、个人资料和营养目标。
- 内置及自定义食物、饮食记录、餐次复制和手动照片留档。
- 内置及自定义动作、训练计划、训练执行和历史记录。
- 体重记录、目标体重和趋势。
- 首页汇总、7/30/90 天统计和周总结。
- 数据导出、恢复、清空和恢复失败回滚。
- Android 签名 APK 构建与离线真机验证。

第一阶段不包含：

- 微信、手机或密码登录，JWT，多用户。
- 远程 API、云存储、跨设备同步。
- 真实或模拟的图片识别。
- 复杂饮食方案、阶段调整和自动膳食计划。
- iOS 安装包。

复杂饮食方案页面在第一阶段应隐藏入口，而不是保留一个无法使用的入口。

## 9. 错误处理与数据安全

- 数据库初始化失败时显示本地数据修复页，支持重试和从备份恢复。
- 保存失败时回滚完整事务，页面保留尚未提交的用户输入。
- 图片复制失败时提示原因，并允许无图片保存。
- 存储空间不足时停止写入并提示清理空间或导出备份。
- 备份导入失败时保持原数据不变，并向用户显示可理解的失败原因。
- APK 升级迁移失败时回滚数据库，不允许使用半迁移数据继续启动。
- 清空数据必须二次确认；清空后重新导入最新种子数据并回到首次设置流程。

## 10. 实施顺序

1. 建立 Android App 构建和 SQLite 基础设施。
2. 迁移首次启动、个人资料和营养目标。
3. 导入食物、动作和训练模板种子数据。
4. 迁移饮食记录和手动照片留档。
5. 迁移体重记录和趋势。
6. 迁移训练计划、执行和历史记录。
7. 迁移首页、统计和周总结业务规则。
8. 实现 ZIP 备份、恢复、恢复点和清空数据。
9. 移除登录、HTTP 请求、失效入口和网络权限。
10. 完成数据库迁移测试、离线真机测试并生成签名 APK。

每一步都应产生可运行、可测试的增量，不进行一次性整体替换。

## 11. 测试策略

- TypeScript 单元测试覆盖营养计算、训练容量、体重趋势、日期范围、周总结和舍入规则。
- Repository 集成测试覆盖 CRUD、外键、事务回滚、软删除、索引查询和 schema 迁移。
- 种子测试验证内置 ID 或稳定键、版本升级及用户数据不被覆盖。
- 备份测试覆盖正常导出恢复、损坏 ZIP、路径穿越、校验失败、不兼容版本、缺失图片和恢复回滚。
- 页面测试验证无登录启动、空状态、保存失败、照片失败和隐藏饮食方案入口。
- Android 真机测试必须在飞行模式下完成首次设置、饮食记录、训练、体重、统计、应用重启、备份和恢复流程。

## 12. 验收标准

- 新安装后无需网络和登录即可完成首次设置并使用核心功能。
- 飞行模式下个人资料、饮食、训练、体重、首页和统计全部可用。
- 应用强制关闭及手机重启后数据仍然存在。
- APK 覆盖升级后已有数据和照片不丢失。
- 清空数据后可以通过导出的 ZIP 完整恢复数据库和用户保留的图片。
- 日、周及 30/90 天统计结果与迁移前后端业务规则一致。
- Android 清单不申请联网权限或访问手机全部文件的权限。
- 第一阶段页面中不存在登录、网络错误、图片识别或未实现复杂饮食方案入口。
- 前端单元测试、SQLite 集成测试和真机离线验收全部通过后，才生成发布签名 APK。

## 13. 后续阶段

第一阶段稳定后，再单独设计并迁移复杂饮食方案、阶段评估和膳食计划。该阶段继续使用同一套 SQLite、Service 和 Repository 边界，不引入服务器依赖。
